## Successes

2. **jest-axe accessibility was a first-class citizen from Story 6.1.** Every component was audited with jest-axe, every story included accessibility tests. This caught the `role="img"` → `role="figure"` issue in DRLifecycleDiagram before UAT. The audit in Story 6.6 found zero new violations — the earlier stories had already been built correctly.

3. **The 10-AC cap continued to work.** Maximum ACs per story was 9 (Stories 6.3, 6.5b). Average review patches per story: 3.0 (comparable to Epic 5's 2.4).

4. **UAT methodology matured further.** The `user-acceptance-test-epic-6.md` document (265 lines) tracked all 9 issues with symptoms, root causes, and fixes. The structured UAT process introduced in Epic 5 carried forward successfully.

5. **DRLifecycleDiagram post-UAT overhaul demonstrates healthy iteration.** Dynamic site names, topology images, and dual transition buttons were applied as polish without a new story — appropriate scope for UX refinement on existing components.

6. **Story 6.5 proactive split was the right call.** 6.5 (lifecycle diagram) had 0 review patches; 6.5b (three content tabs) had 6. Splitting by tab boundaries gave the complex content tabs the attention they needed.

## Challenges

1. **Inaccurate planning documents caused the highest-impact UAT failure.** The architecture, UX, and story specs were based on an assumed future state (OCP 4.22+ with React Router v7, PatternFly 5 tokens) rather than the actual deployment target (OCP 4.20 with React Router v5, PatternFly 6). This mismatch was the root cause of Issue 8 (React Error #130) and required rewriting 7 source files during UAT.

2. **No deployment spike before the epic.** A 30-minute spike deploying a "hello world" plugin to OCP 4.20 would have caught the React Router version, PatternFly version, and HTTPS requirement before any code was written. Instead, these were discovered after 7 stories of development.

3. **Docker/infra issues consumed disproportionate UAT time.** Issues 1–6 (build failures, pull secrets, TLS, document root path) were all infrastructure concerns unrelated to application logic. They required iterative cycles of build → push → deploy → debug that added hours to UAT.

4. **Story 6.5b was still dense despite the split.** Six review patches (the highest in the epic) across three distinct content tabs (Waves, History, Configuration). Each tab was essentially a mini-component. Further decomposition to one-tab-per-story would have reduced review complexity.

5. **Story 4.6 deferred fixes were actually already resolved but not tracked.** All three items (terminal-result annotation cleanup, `resolveVMNamespace` refactoring, annotation dedup) were addressed incidentally during Epic 5 (Stories 5.5–5.7). The deferred-items list became stale because items resolved as side effects of other work were never formally closed.

---

## Key Insights

1. **"Plan for the environment you have, not the one you want."** The React Router v7 assumption was the single biggest source of UAT rework. Planning docs must be validated against the actual deployment target. The retro action is to add an environment validation spike at the start of any epic that introduces a new technology stack.

2. **Unit tests cannot validate webpack module federation or shared module resolution.** Jest runs in Node.js with mock modules. The `Link === undefined` bug could only manifest in a real OCP Console with webpack's shared scope. UAT on a live cluster is non-negotiable for console plugin development.

3. **The PF5 → PF6 transition was less painful than expected.** OCP 4.20 ships PF6, not PF5. Component APIs are largely compatible. The main difference is CSS token naming (`--pf-t--global--*` instead of `--pf-v5-global--*`) and CSS class prefixes (`pf-v6-c-*` instead of `pf-v5-c-*`). Some `--pf-v5-global--*` tokens still resolve as fallbacks.

4. **Navigation placement matters more than it seems.** Issue 9 (plugin under Administration instead of Virtualization) was a UX problem that required inspecting the kubevirt-plugin's actual manifest on the running cluster to discover the correct perspective IDs and section IDs. Planning docs had no guidance on this.

5. **Deferred items need a formal close-out process.** The Story 4.6 items carried across two retros as "not addressed" when they were actually resolved. Process fix: every deferred item must be explicitly verified (resolved or still open) at the start of each epic, not just copied forward.

## Environment Findings Codified

### OCP 4.20 Console Plugin Runtime

| Component | Version | Import Convention |
|-----------|---------|-------------------|
| React | 17.x | `react` (shared scope) |
| React Router | **v5** | `react-router-dom` — `Link`, `useHistory`, `useParams`, `useLocation` |
| PatternFly | **v6** (6.2.3) | `@patternfly/react-core`, `@patternfly/react-table`, `@patternfly/react-icons` |
| PF CSS tokens | `--pf-t--global--*` | PF6 format; `--pf-v5-global--*` still resolves as fallback |
| PF CSS classes | `pf-v6-c-*` | Not `pf-v5-c-*` |
| Console Plugin SDK | 4.20.x | `@console/pluginAPI: "*"` (not `^4.21.0`) |
| Webpack | Module Federation | Shared modules: `react`, `react-router`, `react-router-dom`, `react-router-dom-v5-compat` |

**These are the ground-truth runtime versions. All Epic 7 story specs have been corrected to match.**
