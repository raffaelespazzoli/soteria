# Story 2b.1.5: Cluster-Scoped CRD Migration

Status: done

## Story

As a platform engineer,
I want all Soteria CRDs (DRPlan, DRExecution, DRGroupStatus) to be cluster-scoped,
so that DR plans can unambiguously manage VMs across namespaces and the `soteria.io/drplan` label convention is globally unique by name.

## Background & Motivation

The code review for Story 2b.1 exposed two coupled design defects caused by the mismatch between namespace-scoped DRPlans and their cross-namespace operational intent:

1. **Cross-namespace watch regression:** `enqueueForVM()` enqueues `reconcile.Request{Name: planName, Namespace: vm.Namespace}`, which targets a non-existent object when the DRPlan lives in a different namespace than the VM.
2. **Label ambiguity:** Two DRPlans with the same name in different namespaces collide on the same `soteria.io/drplan` label value, making it impossible for a VM to unambiguously join either plan.

Both defects share a root cause: DRPlan is namespace-scoped but inherently manages resources across namespaces. Making all Soteria CRDs cluster-scoped aligns resource identity with operational scope and resolves both findings.

**Why all three CRDs, not just DRPlan:** DRExecution references a DRPlan by `spec.planName`. DRGroupStatus references a DRExecution by `spec.executionName`. If the parent is cluster-scoped, children must be too — otherwise the parent-child relationship requires a cross-scope reference (namespace + name), which contradicts the simplification goal. All three CRDs are operational platform resources, not tenant artifacts.

**Data migration: not required.** This is a v1alpha1 API with no production data. No migration tooling needed.

## Acceptance Criteria

1. **AC1 — Strategy `NamespaceScoped()` returns `false`:** All three strategy files (`pkg/registry/drplan/strategy.go`, `pkg/registry/drexecution/strategy.go`, `pkg/registry/drgroupstatus/strategy.go`) return `false` from `NamespaceScoped()`.

2. **AC2 — Strategy `GetAttrs` removes `metadata.namespace`:** The `GetAttrs` function in all three strategy files no longer includes `"metadata.namespace"` in the returned `fields.Set`. The `MatchDR*` `SelectionPredicate` functions still work.

3. **AC3 — Reconciler `enqueueForVM` uses empty namespace:** In `pkg/controller/drplan/reconciler.go`, the `enqueueForVM` function enqueues `reconcile.Request{Name: planName, Namespace: ""}` — because DRPlan is now cluster-scoped. This directly resolves review finding #1 from Story 2b.1.

4. **AC4 — ScyllaDB storage continues to work:** The `kv_store` primary key `(api_group, resource_type, namespace, name)` naturally handles cluster-scoped resources with `namespace=""`. `ComponentsToKey` in `keyutil.go` already handles the empty-namespace path. No schema changes needed. Unit and integration tests for storage pass.

5. **AC5 — Admission webhooks work with cluster-scoped plans:** `FindMatchingPlans` in `exclusivity.go` matches by `plan.Name == drplanLabel` without namespace disambiguation — this is now correct because cluster-scoped names are globally unique. `DRPlanValidator` and `VMValidator` validate without namespace references.

6. **AC6 — Integration test CRD stubs are cluster-scoped:** All envtest CRD stubs in `test/integration/*/suite_test.go` use `apiextensionsv1.ClusterScoped` for DRPlan, DRExecution, and DRGroupStatus. VMs remain `NamespaceScoped`.

7. **AC7 — All tests pass:** `make test` and `make integration` pass (excluding pre-existing RBAC failures from Story 2.5). No new lint issues from `make lint`.

## Dev Notes

### Key Insight: Scope Is Controlled by Strategy, Not Type Markers

This project uses an **aggregated API server** (not kubebuilder CRDs). Resource scope is controlled by the `NamespaceScoped() bool` method on strategy files in `pkg/registry/*/strategy.go`, not by `+kubebuilder:resource:scope=` markers. There are no generated CRD YAML files to update.

### ScyllaDB Schema Is Already Compatible

The `kv_store` table primary key is `(api_group, resource_type, namespace, name)`. For cluster-scoped resources, `namespace=""`. The `ComponentsToKey` function in `keyutil.go` already handles this:

```go
func ComponentsToKey(apiGroup, resourceType, namespace, name string) string {
    if namespace == "" {
        return "/" + apiGroup + "/" + resourceType + "/" + name
    }
    return "/" + apiGroup + "/" + resourceType + "/" + namespace + "/" + name
}
```

No ScyllaDB schema migration needed.

### RBAC Implications

Cluster-scoped resources require `ClusterRole`/`ClusterRoleBinding` instead of `Role`/`RoleBinding`. The RBAC integration tests in `test/integration/rbac/` have pre-existing failures (ClusterRole manifests from Story 2.5 were never created). Update the CRD stubs but do not attempt to fix the pre-existing RBAC failures — that's tracked separately.
