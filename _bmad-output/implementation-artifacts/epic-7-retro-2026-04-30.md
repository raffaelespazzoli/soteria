## Successes

1. **100% story completion, zero spawned stories, zero splits.** Cleanest epic delivery in project history — no mid-sprint splits, no organic spawning. Scoping maturity from Epics 4–6 paid off.

2. **Accelerating learning curve within the epic.** Review patches went 4 → 10 → 0 → 1. Story 7.3 was a zero-debug-issues implementation — the first in the project's history. Three stories of accumulated Console SDK + PF6 knowledge reached critical mass.

3. **Module-level singleton pattern for cross-page state.** The `toastStore` architecture in Story 7.4 solved a genuine Console plugin limitation (no shared layout wrapper) with a pure-TypeScript singleton that persists across webpack module federation page navigations.

4. **Shared model extraction happened organically.** Story 7.3 extracted `drExecutionModel` to `k8sModels.ts` when it needed the same model for `k8sPatch` — natural refactoring at the right moment.

5. **UAT methodology continued to mature.** The `user-acceptance-test-epic-7.md` document tracked 8 issues with detailed investigation iterations (Issues 2 and 7 each had 3 attempts). Structured trial-and-error documentation produced reusable platform knowledge.

6. **Accessibility remained a first-class citizen.** Every component audited with jest-axe. ARIA live regions for wave announcements. Screen reader labels on all retry buttons. Zero violations across all states.

## Challenges

1. **Environment validation spike still not executed.** Committed to this in Epic 6 retro. Not done for Epic 7. Three UAT issues (2, 7, 8) were direct consequences — Console SDK behaviors that unit tests cannot reproduce.

2. **Speculative features became dead code.** `estimatedRPO` (built in Epic 5) was never populated by any real driver. `UnprotectedVMs` (built in Epic 5) caused a 30-second status patch loop and UI flicker. Both required cross-stack removal during UAT — deleting code across Go types, controller, metrics, drivers, and 7+ UI components.

3. **Aggregated API server behavior differs from core API expectations.** Issue 8: `MutatingWebhookConfiguration` doesn't intercept requests to aggregated APIs. The fix (using `PrepareForCreate` in registry strategy) was straightforward once understood, but the investigation consumed significant UAT time.

4. **The Console SDK is under-documented for aggregated API patterns.** Issues 2 and 7 are cases where the SDK's documented API (`useK8sWatchResource`, `useParams`) works for core resources but fails silently against aggregated APIs or in plugin page contexts.

5. **UAT is still manual and iterative.** Eight issues, several with 3+ fix attempts. Build → push → deploy → test → debug cycles are time-intensive.

6. **Execution detail page was the hardest component.** Three separate UAT issues (2, 6, 7) stacked on one component — all invisible to the 45 unit tests that passed. Unit test mocks (`useParams`, `useK8sWatchResource`) always return what you tell them; real Console SDK behavior is fundamentally different.

---

## Key Insights

1. **"A validation spike is for interaction patterns, not just technology stacks."** Epic 7 used the same stack as Epic 6, but introduced new SDK interaction patterns (writes, single-resource watches, route params). Each new pattern is a spike candidate.

2. **"Don't build UI for data that doesn't exist."** `estimatedRPO` and `UnprotectedVMs` were both built speculatively in Epic 5 and never backed by real data. They became dead code and caused production bugs. Build UI when data flows end-to-end, not before.

3. **"Unit tests validate logic, not platform integration."** Jest mocks `useParams`, `useK8sWatchResource`, and the Console SDK. Every mock returns what you tell it. Three UAT issues (2, 7, 8) were invisible to all 533 unit tests. UAT on a live cluster remains non-negotiable.

4. **"Accumulated project knowledge reduces review friction to zero."** Story 7.3's clean implementation (0 patches, 0 debug issues) proves that consistent patterns, established conventions, and story intelligence sections produce first-attempt-correct code.

5. **"Cross-stack dead code removal is expensive; prevention is cheap."** Removing `estimatedRPO` touched Go types, controller, metrics, drivers, UI types, hooks, utils, and 7+ UI components. A 5-minute question during planning ("does any driver actually produce this data?") would have prevented all of it.

- **Epic 6:** Console UI — dashboard, plan management, navigation, accessibility
- **Epic 7:** Console operations — pre-flight confirmation, live execution monitor, inline retry, toast notifications
