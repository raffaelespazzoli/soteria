## Manual End-to-End Test (2026-04-21, etl6/etl7 Stretched Clusters)

Full DR lifecycle tested on real infrastructure with Submariner + ScyllaDB multi-DC:

1. **Planned migration** (SteadyState → FailedOver) — VMs stopped on primary, replication stopped, VMs started on secondary
2. **Reprotect** (FailedOver → DRedSteadyState) — storage roles reversed, replication re-established
3. **Failback** (DRedSteadyState → FailedBack) — planned migration back to primary
4. **Reprotect** (FailedBack → SteadyState) — return to original steady state

All four operations completed successfully. Five significant findings surfaced (see Challenges).

## Successes

1. **Full DR lifecycle delivered and validated on real infrastructure.** 8 phases, 4 rest states, all four operations (failover, reprotect, failback, restore) working end-to-end on stretched clusters with real storage (ontap-san via Trident CSI).

2. **Unified FailoverHandler was the architectural highlight.** Story 4.1b absorbed 4.3 and consolidated planned migration and disaster failover into a single handler with different configs. Planned and disaster differ only in Step 0 (graceful shutdown + sync wait vs. skip) and RPO recording — everything else is shared. This eliminated code duplication and made failback a symmetric mirror of failover.

3. **Exceptional test coverage.** ~200+ new tests across the epic, engine coverage at 84.8%. Story 4.7 (checkpoint/resume) alone added 33 tests. The conformance suite from Epic 3 continued to validate driver implementations automatically.

4. **Fail-forward pattern is production-grade.** Story 4.5 introduced per-DRGroup error isolation — a single failed volume group doesn't block recovery of healthy groups in the same wave. `PartiallySucceeded` result with per-group status gives operators actionable detail.

5. **Manual end-to-end test proved the system works.** First time running the full lifecycle on real stretched clusters — this is a milestone. The test surfaced real issues that unit/integration tests could not.

6. **Story specs continued to drive quality.** Consistent output across Opus 4, Opus 4.6, and Sonnet 4.5 — validating the Epic 3 finding that story preparation, not model selection, determines implementation quality.

## Challenges

1. **Eventual consistency flakiness (ScyllaDB optimistic concurrency).** When both etl6 and etl7 controllers run, they can both attempt to update the same DRExecution status in ScyllaDB. Optimistic concurrency conflicts (`the object has been modified`) cause checkpoint status writes to fail, rolling up to `PartiallySucceeded` even when all DR operations succeeded. This is misleading to operators. **Needs monitoring; design fix if pattern persists.**

2. **DRPlan stuck in transient phase after failed execution.** A failed DRExecution left the plan in `FailingBack` with no automatic recovery. `failExecution()` only marks the execution as Failed — it does not roll back the plan phase. Required manual `kubectl replace --raw` on the status subresource. **Resolved: Option C chosen — rest-state-only DRPlan design for Epic 5, Story 5.0.**

3. **Noop driver fallback wasn't wired by default.** The `--noop-fallback` flag controlled it, but the stretched deployment didn't pass the flag. Fixed during the manual test by modifying `noop/driver.go` init() to self-register as the default fallback. The noop-as-catch-all pattern is now always-on.

4. **Storage class misconfiguration in test script.** `DR_TEST_SC` in `hack/stretched-local-test.sh` defaulted to `ocs-storagecluster-ceph-rbd-virtualization` instead of `ontap-san`. Simple config bug but blocked testing. Fixed.

5. **Scope grew to 2 organically spawned stories** (target was 0–1). Story 4.1b (unified FailoverHandler) and 4.9 (site topology fields) were both necessary but indicate the 8-story plan underestimated architectural surface area. The 8-phase, 4-rest-state lifecycle was more complex than a simple handler-per-mode design.

6. **High AC count correlated with lower code precision.** Stories with 13–15 ACs (4.5, 4.6, 4.7, 4.8) showed more review findings and deferred fixes than stories with 7–10 ACs. The LLM context overhead from large AC sets reduces code quality. **Resolved: new 10-AC cap per story.**

7. **Task checkbox maintenance slipped on 2 stories.** Stories 4.6 and 4.9 are marked "done" but still have unchecked task boxes — documentation drift contrary to our Epic 3 commitment.

---

## Key Insights

1. **Manual testing on real infrastructure exposes issues that unit/integration tests cannot.** Eventual consistency, cross-DC controller contention, and deployment config gaps are invisible in envtest. Regular manual testing on stretched clusters should be a recurring practice.

2. **High acceptance criteria count is a code quality anti-pattern for LLM-driven development.** Stories exceeding 12 ACs overwhelm LLM context, producing less precise code with more review findings. The 10-AC cap is a concrete, enforceable process improvement.

3. **The rest-state-only DRPlan model (Option C) eliminates the "stuck phase" bug class by design.** Instead of patching rollback logic into error paths, removing transient phases from the plan makes the system self-healing. Design-level fixes > error-path patches.

4. **Eventual consistency is a feature AND a bug source.** ScyllaDB multi-DC replication enables cross-site state survival, but optimistic concurrency conflicts during concurrent controller writes produce misleading execution results. This tension needs ongoing monitoring and may need a dedicated solution.

5. **Unified handler patterns reduce surface area.** FailoverHandler serving both planned and disaster modes, configured via `FailoverConfig`, is easier to test, easier to review, and produces fewer bugs than separate handlers with duplicated logic.

## Epic 5 Preparation

### New Story: 5.0 — Rest-State-Only DRPlan & Active Execution Pointer

**Rationale:** Manual testing on etl6/etl7 revealed that failed DRExecutions leave DRPlans stuck in transient phases (e.g., `FailingBack`) with no automatic recovery. This is a design flaw: transient phases should not be stored on the DRPlan. The DRPlan should only track rest states, with transient phases derived from the active DRExecution.

**Design Decision (Option C):**
- `DRPlan.Status.Phase` holds only rest states: `SteadyState`, `FailedOver`, `DRedSteadyState`, `FailedBack`
- New `DRPlan.Status.ActiveExecution` field: name reference to in-progress DRExecution (empty when idle)
- `EffectivePhase()` helper derives transient phase from rest state + active execution mode
- Reconciler sets `ActiveExecution` on start, clears on completion/failure
- Phase advance to next rest state on success or partial success; no advance on total failure (plan stays at current rest state)
- Admission webhook rejects new execution if `ActiveExecution != ""`
- Printer column shows computed effective phase for `kubectl get drplans`
- All test fixtures updated — plan never holds transient phases

**Sequencing:** Story 5.0 runs before Story 5.1. The phase model must be settled before monitoring stories depend on plan status.
