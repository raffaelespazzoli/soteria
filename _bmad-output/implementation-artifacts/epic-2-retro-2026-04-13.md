2. **Organic story spawning process validated again.** Story 2.3.1 (VM Label Validation Webhook) was spawned from Story 2.3's gap analysis, following the same pattern as Story 1.3.1 in Epic 1. The process for catching gaps and creating proper stories is mature and repeatable.

3. **Documentation standards fully adopted.** Every story explicitly verifies Tier 1/2/3 documentation standards established in the Epic 1 retrospective. Package godocs, architecture block comments, and domain 'why' comments were consistently applied across all new code.

4. **Code reviews catching real defects.** Story 2.4: always-populate preflight on every exit path + `+listType=atomic` markers. Story 2.5: `apierrors.IsNotFound()` error classification, `ErrAmbiguousSource` guard. Story 2.3.1: kubebuilder markers + test isolation fixes. These are correctness issues, not style nits.

## Challenges

1. **Story 2.2 task checkboxes remain unchecked despite "done" status.** The story file shows `[ ]` on every task while the Dev Agent Record says "done." Task-level tracking in the story file wasn't maintained during implementation.

2. **Deferred items accumulating.** Story 2.5 added 3 new deferred review findings (sanitization wiring, output tests, aggregated API RBAC testing). Combined with 2 items carried from Epic 1, the deferred backlog is now 7 items. All are legitimately deferred (blocked or low priority), but the pattern of growing deferral needs attention.

3. **Static driver map is a bridge.** Story 2.4's storage backend resolver uses a hardcoded `StorageClassDriverMap` with only `"noop"` recognized in production. All real storage classes resolve as `"unknown"` until Epic 3. Preflight reports are of limited production value until then.

4. **Webhook deployment caveat.** Story 2.3's `ValidatingWebhookConfiguration` with `matchPolicy: Equivalent` for aggregated API resources may not work on all Kubernetes versions. A fallback (in-process admission plugin) is documented but not implemented — a deployment risk for when the operator ships to real clusters.

---

## Key Insights

1. **Event-driven controller patterns scale well.** The DRPlan controller's secondary VM watch with `mapVMToDRPlans` and `vmRelevantChangePredicate` gives sub-second response to VM changes while avoiding unnecessary reconciliation. The 10-minute fallback requeue is a good safety net.

2. **Defense-in-depth validation works.** The two-layer validation (admission webhooks for cross-resource checks + strategy-level for per-object field checks) provides robust error prevention. Even if a webhook is temporarily unavailable, the aggregated API server's strategy catches basic errors.

3. **Aggregated API + controller-runtime in a single binary is viable.** The pattern established in Epic 1 scaled smoothly to controller reconciliation, webhook serving, and status updates. No architectural surprises.

4. **Typed kubevirt dependency was the right call.** Direct `kubevirt.io/api` dependency gives compile-time safety and clean VM spec access. The `VMDiscoverer` interface still allows test mocks while the production path benefits from typed access.
