## Successes

1. **100% story completion with zero organic story spawning.** All 8 stories delivered. Stories 5.5 and 5.6 were design corrections identified from E2E testing — they addressed engine correctness, not scope creep. Planning accuracy improved from Epic 4 (2 spawned) to Epic 5 (0 spawned).

2. **Write-contention fix is a genuine architectural win.** The 4-pronged solution (MergeFrom patch, setup yield, ScyllaRetry backoff, fetchPlanWithActiveExecCheck) reduced ERROR logs from 34+ per execution to 2 benign ones. This eliminates an entire class of misleading `PartiallySucceeded` results — the #1 concern from Epic 4's retro.

3. **Story 5.0's self-healing design eliminated the "stuck transient phase" bug class.** When an execution fails, `failExecution` clears `ActiveExecution` and the plan returns to its rest state. No more `kubectl replace --raw` for stuck plans.

4. **The UAT process matured into a structured validation methodology.** Six runs with documented observations, error counts, and regression checks. The `user-acceptance-test-epic-5.md` file (700+ lines) tracks the entire testing journey — from first bug to final clean run.

5. **10-AC cap improved code precision.** Max ACs dropped from 15 (Epic 4, Story 4.6) to 9. Average review patches per story dropped from ~3.6 (Epic 4) to ~2.4 (Epic 5).

6. **Site-aware ownership (5.5) resolved the cross-site contention that plagued every E2E test since Epic 4.** Each controller now knows its role — Owner, Step0Only, or None — and the ownership model is derived from the state machine, not hardcoded.

7. **Story 5.7 simplified the driver interface from 7 to 6 methods and unified the per-group failover path.** `SetTarget` removal was a net reduction of 684 lines. `Force` flags removed. Both planned and disaster modes now use `StopReplication → StartVM`.

## Challenges

1. **ScyllaDB eventual consistency was the single biggest time sink.** It drove Stories 5.5, 5.6, and the entire write-contention investigation (Runs 4–6). `retry.DefaultRetry` with 10ms/5-step backoff was designed for etcd, not ScyllaDB. Four distinct manifestations: `ensurePlanNameLabel`, `persistStatus`, `KubeCheckpointer.WriteCheckpoint`, and the stale execution guard.

2. **Three code bugs that unit tests couldn't catch.** Run 1: `persistStatus` condition overwrite. Run 1: Step0Complete bypass. Run 3: `SetSource` vs `StopReplication` mismatch. All required live multi-site testing. envtest runs a single controller against a single apiserver — multi-controller concurrency is fundamentally untestable there.

3. **Dev Agent Record sections empty for 4 of 8 stories (5.0, 5.5, 5.6, 5.7).** Root cause: these stories were implemented reactively during UAT sessions, bypassing the `bmad-dev-story` workflow. The finalization step that populates the record never ran. **Resolved: records backfilled during this retrospective.**

4. **Task checkbox drift continued for a third epic.** Stories 5.5, 5.6, and 5.7 have unchecked task boxes despite being done. Same pattern as Epics 3 and 4. The manual checkbox process does not survive reactive/hotfix implementation modes.

5. **Story 4.6 deferred fixes still unaddressed.** Terminal-result annotation cleanup, `resolveVMNamespace` ctx propagation, and duplicate annotation dedup remain open from Epic 4.

## Key Insights

1. **Manual E2E testing found design-level issues, not just bugs.** Run 1 drove Story 5.5 (site-aware ownership was entirely missing) and Story 5.6 (wave gate was missing). The E2E test essentially drove 2.5 stories of work that wasn't planned. Unit tests catch logic errors; E2E tests catch architectural gaps.

2. **ScyllaDB eventual consistency is a systemic architectural concern, not a per-bug issue.** Every Kubernetes controller pattern assumes etcd with strong consistency. `retry.DefaultRetry`, the standard informer cache, and read-after-write patterns all assume the write is visible on the next read. With ScyllaDB, it's not. Any future controller code must be written with this awareness — codified as `ScyllaRetry` and `MergeFrom` patterns in the architecture doc.

3. **The 10-AC cap is validated.** Smaller AC sets (max 9 vs max 15) produced tighter implementations with fewer review findings. The cap should remain permanent.

4. **Stories implemented outside `bmad-dev-story` workflow lose their documentation trail.** The 4 stories with empty Dev Agent Records were all implemented reactively. Process fix needed: either enforce finalization for all stories regardless of workflow, or create a standalone backfill skill.

5. **The progression from Run 1 to Run 6 demonstrates genuine engineering rigor.** Each run characterized a problem, a fix was applied, and the next run validated it. This is the UAT methodology we should carry forward.

## Significant Discovery: ScyllaDB Eventual Consistency Patterns

Epic 5 produced a set of operational patterns for working with ScyllaDB eventual consistency that are now codified in the architecture document:

| Pattern | Purpose |
|---------|---------|
| `engine.ScyllaRetry` | Tuned retry backoff (200ms/2.0/0.3/8 steps) replacing `retry.DefaultRetry` |
| `client.MergeFrom` patch | Strategic merge patch reducing conflict surface vs full `client.Update` |
| Setup yield | `RequeueAfter: 1ms` after initial writes to force fresh object fetch |
| `fetchPlanWithActiveExecCheck` | Consistency-safe DRPlan fetch with retry loop for ActiveExecution |
| Checkpoint backoff alignment | `KubeCheckpointer` uses ScyllaRetry-aligned parameters |

**Any future controller code that does `Get → modify → Update` against ScyllaDB-backed resources must use these patterns.**
